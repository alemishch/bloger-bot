# Техническое задание — Bloger Bot

## 1. ОБЩЕЕ ОПИСАНИЕ ПРОЕКТА

### 1.1 Цель проекта

Разработка двух цифровых помощников на базе методологии экспертов (Юрий Кинаш, Мария Кинаш) в формате Telegram-бота с мини-приложением (Telegram Mini App) для персонализированной диагностики, составления стратегии и ежедневного сопровождения пользователей с хроническими симптомами. Боты работают по отдельности и являются двумя независимыми продуктами.

### 1.2 Целевая аудитория

- **Основной сегмент:** Женщины 30-45 лет с хроническими симптомами (усталость, боли, ЖКТ, лишний вес)
- **Второй сегмент:** Лояльные клиенты эксперта после прохождения курсов с частичным результатом (30-80%)

### 1.3 Ключевая особенность

Система реализует два независимых Telegram-бота — для Блогера №1 (Юрий) и Блогера №2 (Мария). Каждый бот имеет собственный Telegram-токен, отдельную базу контента и уникальный дизайн. Оба работают на единой кодовой базе и общей серверной инфраструктуре (white-label архитектура), что позволяет масштабировать решение на новых блогеров без переписывания кода.

### 1.4 Путь клиента (воронка)

1. Instagram (ссылка в сторис/рилс, ключевое слово в директ) → переход в Telegram
2. Онбординг в боте → первичные вопросы (тема, что беспокоит, что пробовал) + получил контент по теме (лидмагнит)
3. Переход в Mini App → диагностика (базовый профиль + вопросы от LLM)
4. Получение резюме профиля и персональной стратегии и контенту (3-4 видео на странице с профилем)
5. Оффер подписки → оплата → доступ к полному функционалу

## 2. АРХИТЕКТУРА СИСТЕМЫ

### 2.1 Общая структура

**Тип:** Микросервисная архитектура

**Микросервисы системы:**

- **Telegram Bot Service** — обработка сообщений, команд, webhook (отдельный экземпляр для каждого блогера)
- **Web App Service** — Telegram Mini App / WebView (отдельный экземпляр для каждого блогера)
- **LLM Service** — RAG, обработка запросов к языковой модели, Tone of Voice
- **Content Service** — управление контентом, поиск по базе знаний, векторная БД
- **User Service** — управление пользователями, профилями, долгосрочной памятью
- **Analytics Service** — сбор метрик, аналитика, интеграция с Metabase
- **Payment Service** — обработка платежей, рекуррентные списания, управление подписками
- **CRM Integration Service** — интеграция с AmoCRM
- **Proactive Ping Service** — микросервис проактивных пингов
- **Frontend**

Возможно добавление или изменение списка микросервисов при выявлении необходимости во время разработки.

### 2.2 Инфраструктура

**Хостинг:**
- Требуется аккаунт для хостинга (VPS/Dedicated или облако)
- Оплата хостинга: на стороне заказчика
- Требования к серверу:
  - Минимум 4 CPU cores
  - Минимум 8GB RAM
  - Минимум 100GB SSD
  - Пропускная способность: минимум 100 Mbps
  - Доступ: SSH, root права
- Варианты размещения: собственные серверы, YandexCloud, VK Cloud (выбор согласовывается)

**База данных:**
- Основная БД: PostgreSQL (для структурированных данных)
- Векторная БД: ChromaDB / Qdrant / pgvector (для RAG)
- Кеш: Redis (для сессий, временных данных)

**Резервирование:**
- Репликация БД (master-slave)
- Автоматическое резервное копирование (ежедневно)
- Healthchecks и автоматический перезапуск сервисов

**Мониторинг:**
- Grafana — визуализация метрик
- Prometheus — сбор метрик
- Алерты: критические ошибки, падение сервисов, проблемы с оплатами

## 4. ФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ

### 4.1 Онбординг

**Сценарий:**
1. Пользователь запускает бота командой /start
2. Приветственное сообщение с описанием возможностей
3. Правовая рамка: "не медицинский диагноз, не замена терапии"
4. Проверка в AmoCRM: система проверяет наличие контакта по номеру телефона
   - Если контакт существует → подгружает данные профиля
   - Если контакт новый → создаёт новый контакт в AmoCRM
5. Переход к регистрации или диагностике

**Данные для сбора:**
- Телефон (через Telegram)
- ФИО
- Email (опционально)

**Интеграция с AmoCRM:**
- API: AmoCRM REST API v4
- Действия:
  - Поиск контакта по номеру телефона
  - Создание нового контакта (если не найден)
  - Синхронизация данных профиля
  - Создание сделки при оформлении подписки
  - Обновление статусов и тегов
- Триггеры:
  - Регистрация → создание контакта
  - Прохождение диагностики → добавление тега
  - Оформление подписки → создание сделки
  - Отмена подписки → обновление статуса

### 4.2 Диагностика

**Этап 1: Статическая анкета**
- Формат: кнопки выбора, текстовые поля
- Вопросы:
  - Основные симптомы (усталость, боли, ЖКТ, сон, вес)
  - Что уже пробовали
  - Образ жизни (работа, семья, дети)
  - Что мешает изменениям + соцдем, семейное положение

**Этап 2: Уточняющие вопросы от LLM**
- 2-4 динамических вопроса на основе ответов
- Цель: выявить паттерны по методологии эксперта
- Формат: диалог с ботом

Вопросы статической анкеты и логика уточняющих вопросов от LLM разрабатываются на основе методологии эксперта. Перечень вопросов и критерии интерпретации ответов передаются командой Заказчика отдельно для каждого блогера (Юрий Кинаш — Блогер №1, Мария — Блогер №2) до начала разработки соответствующего этапа.

**Результат:**
- Формирование "карты проблемных зон"
- Структура: тело/медицина, психика, образ жизни, глубинные истории

### 4.3 Персональная стратегия

**Генерация плана:**
- Срок: 1-3 месяца (настраивается)
- Содержание:
  - 1-2 приоритетных направления
  - Что делать самому
  - Что обсудить с врачом
  - Что вынести в терапию

3-5 варианта (количество вариантов индивидуально генерируется) направления (как гипотезы), что нужно изучить в себе — ветки, и план как по ним идти

\+ тезис с оценкой человека по зонам (по колесу баланса)

Шаблон ответа (структура, логика создания) — Создается Юрием и командой Эксперта

Шаблон ответа (структура, логика создания) разрабатывается командой эксперта отдельно для каждого блогера и передаётся Исполнителю до начала Этапа 2. Исполнитель реализует генерацию стратегии строго в рамках переданных шаблонов. Содержательная ответственность за методологию стратегии лежит на стороне Заказчика.

**Подбор контента:**
- Автоматический подбор релевантного контента эксперта
- Типы: тексты, видео, аудио
- 1-3 материала на каждую проблемную зону

### 4.4 Основной функционал продукта

**Структура Mini App (3 блока):**

**Блок 1: AI Юра (центральный)**
- Переосознание дня: цитата/метафора по теме пользователя
- Задать вопрос цифровому эксперту
- Уведомления о вебинарах
- Ежедневные задания (контент, практики)
- Единица контента из базы (по актуальной проблеме)
- Рубрика "Ответы Юры пользователям" (Q&A)

**Блок 2: Твоя стратегия (левый)**
- Персональная стратегия на период
- Личный прогресс (колесо баланса, графики)
- Кнопка "Служба заботы"
- Кнопка "Отмена подписки" (не явная)

**Блок 3: Библиотека (правый)**
- Сохраненное (контент, тезисы из диалогов)
- Библиотека по темам:
  - Бесплатные темы (для всех)
  - Сохраненное из диалогов
  - Медитации
  - Практика месяца (открывается каждый месяц новая)
  - Записи эфиров/конспекты
  - Задать вопрос для эфира

Контент блоков (цитаты, метафоры, Q&A) предоставляется командой каждого эксперта отдельно для Блогера №1 (Юрий) и Блогера №2 (Мария). Исполнитель обеспечивает техническую реализацию отображения и ротации контента.

### 4.5 Ежедневные задания

**Механика:**
- Две дорожки: "здоровье и тело" + "состояние и психика"
- Генерация заданий на день/неделю
- Напоминания (настраиваемое время)
- Чек-листы выполнения

На день, 1 задание

**Сценарий "после эксперта":**
- Отдельная логика для тех, кто уже проходил курсы
- Фокус на закреплении и интеграции

База заданий по обоим дорожкам («здоровье и тело» + «состояние и психика») составляется командой эксперта отдельно для каждого блогера (Юрий Кинаш — Блогер №1, Мария — Блогер №2) и передаётся Исполнителю до начала Этапа 3. LLM персонально адаптирует последовательность и формулировки заданий под профиль пользователя — но не генерирует задания с нуля. Содержательная ответственность за задания лежит на стороне Заказчика.

### 4.6 Трекинг прогресса

**Визуализация:**
- Колесо баланса / карта зон (динамическая)
- Дни активности (streak)
- Динамика состояния (графики)
- Выполненные задания

Колесо баланса с метриками + визуализация прогресса в авторской форме (опционально — сделать визуализацию по запросу в чате)

**Данные для отслеживания:**
- Ежедневные шкалы самочувствия
- Выполнение заданий
- Активность в боте
- Изменения по зонам

Структура зон (тело/медицина, психика, образ жизни, глубинные истории), критерии наполнения и визуальная логика колеса баланса определяются методологией эксперта и передаются командой Заказчика отдельно для каждого блогера до начала разработки соответствующего этапа. Исполнитель реализует визуализацию и динамику на основе переданных критериев.

### 4.7 Чат "Разбор ситуации"

**Функционал:**
- Описание текущей ситуации пользователем
- LLM уточняет контекст
- Привязка к карте зон и стратегии
- Предложение 1-2 реалистичных шагов

**Требования к качеству:**
- Ответ в тоне эксперта (tone of voice)
- Использование контента из базы знаний
- Цитирование или переформулирование (не прямое копирование)
- Среднее время ответа: ~10 секунд (реалистичная оценка с учётом RAG)

**UX во время ожидания ответа (10 секунд):**

Проблема: 10 секунд — это долго для пользователя, есть риск, что он уйдёт или подумает, что бот завис

Решение — Механика удержания внимания:

*Индикаторы прогресса (обязательно):*
- Анимированный индикатор загрузки
- Текстовые статусы процесса:
  - "Изучаю вашу ситуацию…" (0-3 сек)
  - "Подбираю релевантный опыт…" (3-6 сек)
  - "Формирую рекомендации…" (6-10 сек)

*Интерактивные элементы (желательно):*
- Микро-факты о здоровье (показываем случайный)
- Анимация "думающего Юры" (аватар с анимацией)
- Прогресс-бар с процентами

*Технические оптимизации:*
- Показывать статус "печатает…" сразу при отправке вопроса
- Streaming ответа (показывать по мере генерации, если возможно)
- Кеширование частых запросов для ускорения

**Приоритет для MVP:** Как минимум статусы процесса + анимация загрузки

### 5.3 Разметка контента

Процесс разметки и ответственность сторон: Исполнитель выполняет первичную автоматическую разметку контента с помощью LLM. Результат экспортируется в таблицу (Excel/CSV/или другой формат) и передаётся Заказчику для ручной верификации и уточнения — в первую очередь по тегам тематики, проблемных зон и типа контента. Качество итоговой RAG-системы напрямую зависит от качества ручной проверки: без участия эксперта со стороны Заказчика точность подбора контента и корректность ответов LLM могут быть ниже ожидаемых. Срок верификации каждого батча — не более 5 рабочих дней с момента передачи; задержка сверх этого срока продлевает сроки этапа.

**Метаданные для каждого материала:**
```json
{
  "id": "unique_id",
  "type": "video|audio|text|post",
  "source": "youtube|telegram|webinar",
  "title": "Название",
  "description": "Краткое описание",
  "duration": "хронометраж",
  "transcript": "полная транскрипция",
  "summary": "краткое содержание",
  "tags": ["боль", "жкт", "психосоматика"],
  "themes": ["здоровье", "психика", "образ жизни"],
  "problems_solved": ["усталость", "боли в животе"],
  "tools_mentioned": ["дыхательные практики", "гипноз"],
  "target_audience": "новички|продвинутые|после_курса",
  "content_type": "serious|humor|metaphor",
  "is_paid": false,
  "author": "yuri|maria"
}
```

**Процесс разметки:**
1. Автоматическая разметка LLM
2. Экспорт в таблицу для верификации
3. Ручная проверка и корректировка экспертом
4. Импорт размеченных данных в систему

**Критический риск — Шутки и юмор спикера:**

Решение: Конкретный подход к обработке юмора и метафор будет определён в процессе разработки — заранее невозможно знать оптимальный вариант. Один из рассматриваемых вариантов — включение инструкций по интерпретации юмора и метафор в систему Tone of Voice через примеры ответов (few-shot промптинг): LLM учится отличать буквальный контент от образного на конкретных примерах из материалов эксперта. Тегирование на уровне фрагментов не применяется, так как текст разбит крупными смысловыми блоками, а не предложениями — добавление гранулярных тегов нарушило бы требование по времени ответа ≤10 сек. Справочник «проблемных фраз» из ТЗ исключается как избыточный.

**Хэштеги/теги:** Формирование справочника тегов (30–50 тегов) осуществляется командой Исполнителя совместно с Заказчиком. Первичное присвоение тегов производится автоматически (LLM), после чего таблица с разметкой передаётся Заказчику для ручной верификации и корректировки. Итоговое качество разметки зависит от полноты и своевременности ручной проверки со стороны Заказчика.

### 5.4 База знаний (RAG)

**Структура:**
- Векторизация контента (embeddings)
- Хранение в векторной БД
- Индексация по тегам, темам, проблемам
- Связь с метаданными

**Поиск:**
- Семантический поиск по embeddings
- Фильтрация по тегам и метаданным
- Гибридный поиск (векторный + полнотекстовый)

## 6. LLM И RAG

### 6.1 Модель

**Выбор модели:**
- Основная: OpenAI GPT-4 / GPT-3.5-turbo (или аналоги)
- Выбор между моделями в зависимости от сценария:
  - GPT-4: сложные запросы, диагностика, генерация стратегии
  - GPT-3.5-turbo: простые вопросы, подбор контента

Ориентировочная стоимость API на одного активного подписчика: ~400 000 — 1 500 000 токенов в месяц (input + output суммарно), что составляет приблизительно 400 рублей в месяц на подписчика при текущих тарифах OpenAI. Данная оценка является ориентировочной и зависит от интенсивности использования. Расходы на API несёт Заказчик отдельно от стоимости разработки.

**Требования к доступу и бюджету:**

*Аккаунт ChatGPT/OpenAI:*
- Нужен аккаунт с API ключами для разработки
- Доступ к дашборду для мониторинга расхода токенов
- Возможность отслеживания usage в реальном времени

*Бюджет на разработку:*
- Транскрибация (Whisper): ~$70 (на весь объём контента)
- Токены GPT на разработку: заложить бюджет на тестирование и отладку
- Токены на разметку контента: автоматическая разметка через API
- Рекомендуется: $200-300 на весь период разработки

### 6.2 RAG (Retrieval-Augmented Generation)

**Архитектура:**
1. Получение запроса пользователя
2. Генерация embedding запроса
3. Поиск релевантного контента в векторной БД
4. Формирование контекста для LLM
5. Генерация ответа с использованием контекста

**Векторная БД:** ChromaDB (простота) / Qdrant (производительность) / pgvector (интеграция с PostgreSQL)

**Параметры:**
- Top-K: 3-5 релевантных документов
- Similarity threshold: 0.7-0.8
- Max context length: 4000-8000 токенов

### 6.3 Tone of Voice

**Подход:**
- Промпт-инжиниринг с примерами (few-shot)
- Система промптов с описанием манеры общения эксперта
- НЕ дообучение (fine-tuning) — слишком дорого и долго

**Реализация:**
1. Сбор примеров текстов эксперта (10-20 примеров)
2. Выделение характерных фраз, манеры
3. Формирование system prompt
4. Итеративная проверка и корректировка

### 6.4 Метрики качества

**Тестирование:**
1. Создание тестовой базы вопросов-ответов (50-100 пар)
2. Тестирование на фокус-группе (20-30 человек)
3. Сбор фидбека
4. Экспертная верификация ответов
5. Итеративное улучшение

**Целевые показатели:**
- Релевантность ответов: 70%+ (MVP) / 80%+ (полная версия)
- Tone of voice: соответствие оценке эксперта
- Время ответа: 2-5 секунд (до 10 приемлемо)

Релевантность измеряется на тестовой базе вопросов-ответов (50–100 пар), составленной командой Заказчика совместно с экспертом до начала тестирования. Каждый ответ оценивается по бинарной шкале («релевантен / не релевантен») ответственным лицом со стороны Заказчика. Вопросы должны быть по темам, которые обработаны и внесены в систему. Процент релевантных ответов от общего числа = итоговая метрика. Оценка проводится на этапе приёмки каждой версии и итеративно в процессе разработки по согласованию сторон.

### 7.1 Причины использования Mini App

- Обеспечивает пользовательский опыт, соответствующий уровню платного продукта

### 7.2 Функционал Mini App

**Экраны:**
1. **Главная** — прогресс, задания на сегодня. Задания формируются на основе базы заданий, подготовленной командой эксперта; LLM адаптирует их персонально под профиль пользователя (не генерирует задания с нуля).
2. **Диагностика** — прохождение анкеты
3. **Карта зон** — визуализация проблемных зон. Визуализация строится по методологии эксперта; структура зон (тело/медицина, психика, образ жизни, глубинные истории) и критерии их наполнения передаются Заказчиком до начала разработки соответствующего этапа.
4. **Стратегия** — план на месяц/период
5. **Задания** — список текущих и выполненных. Задания выстраиваются на основе готовой базы от команды эксперта.
6. **Прогресс** — графики, streak, динамика. LLM персонально адаптирует последовательность и формулировки. Прогресс отображается в едином фреймворке с картой зон: динамика строится на основе просмотренного контента, выполненных заданий и самооценок самочувствия из контекста диалогов.
7. **Контент** — библиотека материалов эксперта
8. **Чат** — интерфейс диалога с ботом
9. **Профиль** — настройки, подписка, данные

### 7.3 Дизайн

**Требования:**

- Адаптивность (mobile-first)

## 12. РИСКИ И ОГРАНИЧЕНИЯ

### 12.1 Технические риски

**Качество разметки контента**
- Риск: большой объём ручной работы
- Митигация: итеративная проверка на малом датасете

**Шутки и юмор спикера**
- Риск: некорректная интерпретация LLM, репутационные потери
- Митигация: подход определяется в процессе разработки; один из рассматриваемых вариантов — инструкции по интерпретации юмора и метафор через few-shot промптинг в системе Tone of Voice. Тегирование на уровне отдельных фрагментов не применяется.
- Приоритет: ВЫСОКИЙ

**Точность ответов LLM**
- Риск: галлюцинации, нерелевантные ответы
- Митигация: RAG, промпт-инжиниринг, тестирование

**Производительность RAG**
- Риск: медленный поиск при большом объёме контента
- Митигация: оптимизация индексов, кеширование

**Стоимость API**
- Риск: превышение бюджета на токены
- Митигация: мониторинг расхода, лимиты на пользователя

### 12.2 Продуктовые риски

**Неясность сценариев**
- Требуется детальный user flow
- Примеры диалогов
- Конкретные требования к логике

**Объём контента**
- 800+ страниц контента требует времени
- Риск затягивания разметки

- Релевантность ответов: 80%+ на тестовой базе из 50–100 вопросов. Методика измерения: каждый вопрос в тестовой базе сопровождается списком допустимых источников из базы знаний; ответ считается релевантным, если он опирается хотя бы на один из указанных источников. Оценка производится ответственным лицом со стороны Заказчика по бинарной шкале.
- Время ответа LLM: < 10 секунд
- Uptime: 99%+
- Конверсия в оплату: тестируется
- Retention D7: тестируется

## 14. ДОЛГОСРОЧНАЯ ПАМЯТЬ И ПРОФИЛЬ ПОЛЬЗОВАТЕЛЯ

### 14.1 Архитектура памяти

Продукт использует двухуровневую систему памяти: "горячая" история последних двух сессий и "холодный" JSON-профиль с долгосрочным саммари по пользователю. Это позволяет LLM помнить контекст недавних разговоров и при этом не перегружать контекстное окно.

### 14.2 Определение сессии

Сессия — это непрерывный диалог пользователя с ботом. Сессия считается завершённой, если с момента последнего сообщения прошло более 2 часов. Хранятся последние 2 завершённые сессии в сыром виде (история сообщений). Сессии старше двух последних не хранятся в сыром виде — их содержание попадает в долгосрочный профиль через саммари.

### 14.3 Обновление профиля

После завершения каждой сессии автоматически запускается агент-апдейтер. Он анализирует диалог и принимает решение: обновить соответствующие поля профиля или не менять ничего, если в сессии не было новой значимой информации. Агент не дописывает данные механически — он должен явно решить, что изменилось. Обновление происходит асинхронно после завершения сессии, не во время диалога.

### 14.4 Структура JSON-профиля

Ограничение на размер профиля: 4 000 символов. При обновлении агент обязан сжимать устаревшие данные, а не дописывать поверх.

Поля профиля выбраны с расчётом на то, что LLM среднего уровня сможет надёжно извлечь их из диалога без домысливания:

- **name** — как пользователь просит себя называть (только если явно назвал себя в диалоге)
- **communication_style** — формально/неформально, с юмором или без, предпочитает короткие или развёрнутые ответы. Выводится из паттерна самой переписки, не со слов пользователя
- **goals** — чего хочет достичь, зачем пришёл в продукт. Только если пользователь говорил об этом явно
- **completed_lessons** — список пройденных занятий и курсов. Берётся из системы (факт), агент не выводит это из диалога
- **topics_of_interest** — темы, которые пользователь поднимал в диалогах (ЖКТ, сон, тревога и т.д.)
- **reactions** — что заходило хорошо, что вызывало негатив или игнор. Только явные сигналы: пользователь написал "это не про меня", закрыл тему, вернулся к ней снова и т.д. Агент не интерпретирует молчание как реакцию
- **last_session_summary** — 3–5 предложений о чём была последняя сессия
- **previous_session_summary** — то же для предпредыдущей сессии

Намеренно не включаются в профиль: возраст, профессия, эмоциональное состояние — LLM вероятно будет домысливать эти данные, что создаёт риск ошибок.

## 15. ПРОАКТИВНАЯ КОММУНИКАЦИЯ

### 15.1 Принцип

Проактивные пинги — это персональные сообщения от бота, которые генерируются на основе профиля пользователя и истории его сессий. Это не шаблонные push-уведомления, а сообщения, содержательно связанные с тем, о чём конкретный человек общался и что для него актуально.

### 15.2 Механика микросервиса

Сервис проактивных пингов работает по следующей логике:

1. **Триггер.** Запуск раз в N дней (настраивается в конфигурации, дефолт — каждые 3 дня). Дополнительный триггер — если пользователь не открывал бота более 5 дней.
2. **Входные данные для LLM.** JSON-профиль пользователя + сырая история последних двух сессий.
3. **Задача LLM.** Решить: есть ли смысл пинговать этого пользователя прямо сейчас? Если да — сгенерировать персональное сообщение. Если нет — вернуть флаг `skip: true` без сообщения. Примеры поводов для пинга: пользователь упоминал, что хочет разобраться с темой X — напомнить и предложить следующий шаг; пользователь давно не возвращался к своей стратегии; пройдено новое занятие — предложить закрепление.
4. **Канал доставки.** Сообщение в Telegram-чат с ботом (не email, не push).
5. **Ограничения.** Не более 1 пинга в 3 дня на пользователя. Пользователь может отключить пинги через настройки.

### 15.3 Формат ответа LLM

LLM возвращает JSON следующей структуры:
```json
{ "skip": false, "message": "текст персонального сообщения" }
```
или
```json
{ "skip": true }
```
При `skip: true` сообщение не отправляется и следующая попытка запланируется через N дней.
